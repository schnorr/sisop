* Mutex (Mutual Exclusion)

Um mutex é uma variável compartilhada: destravado (0) ou travado (!=0)
- Pode ser empregado um único bit, mas tipicamente é um valor inteiro
- Podemos implementar com a instrução TSL (veja aula passada)
- É a ferramenta mais simples para garantir exclusão mútua

#+latex: \vfill

Funcionamento
- Processo deve adquirir o lock antes de entrar na seção crítica
- Processo libera o lock quando sai da seção crítica

#+latex: \vfill

#+begin_src C
/* função para entrar na região crítica */
acquire() {
  while (!available); /* espera ativa (spinlock) */
  available = false;
}

/* função para sair da região crítica */
release() {
  available = true;
}
#+end_src

#+latex: \vfill

- =acquire()= e =release()= devem ser executadas atomicamente
- Principal desvantagem: espera ativa desperdiça ciclos de CPU
- Spinlocks são úteis quando o lock é mantido por períodos curtos

* Mutex com =thread_yield= (evitando a espera ativa)

Podemos evitar a espera ativa com =thread_yield=
- Chama o escalonador de threads no espaço do usuário
#+latex: \vfill

#+begin_src C
mutex_lock:
    TSL REGISTER,MUTEX  ; copia mutex e atribui 1
    CMP REGISTER,#0     ; o mutex era zero?
    JZE ok              ; era zero: mutex livre, retorna
    CALL thread_yield   ; mutex ocupado: escalone outra thread
    JMP mutex_lock      ; tente novamente
ok: RET

mutex_unlock:
    MOVE MUTEX,#0       ; coloca 0 em mutex
    RET
#+end_src

#+latex: \vfill

- =mutex_lock= e =mutex_unlock= não exigem chamadas ao núcleo
- Threads de usuário sincronizam inteiramente no espaço do usuário
  - Evitam chamadas ao escalonador do kernel

* Mutex em Pthreads

Pthreads proporcionam funções para sincronizar threads via mutex

- Uma thread trava o mutex para entrar na região crítica
- Se o mutex já estiver travado, a thread que chamou é bloqueada
- Quando desbloqueada, uma thread esperando é autorizada a continuar

#+latex: \vfill

| Chamada de thread       | Descrição                      |
|-------------------------+--------------------------------|
| =pthread_mutex_init=      | Cria um mutex                  |
| =pthread_mutex_destroy=   | Destrói um mutex existente     |
| =pthread_mutex_lock=      | Obtém uma trava ou é bloqueado |
| =pthread_mutex_trylock=   | Obtém uma trava ou falha       |
| =pthread_mutex_unlock=    | Libera uma trava               |

#+latex: \vfill

Comentários
- =pthread_mutex_trylock=: permite espera ocupada quando necessário
- Mutexes podem ter atributos para fins especializados
- As travas não são obrigatórias; cabe ao programador usá-las corretamente

* Exemplo de Mutex com Pthreads

#+begin_src C :tangle aula09-mutex.c
#include <stdio.h>
#include <pthread.h>

#define NTHREADS 4
#define NINC 1000000

long contador = 0;
pthread_mutex_t mutex;

void *incrementa(void *arg) {
  for (long i = 0; i < NINC; i++) {
    pthread_mutex_lock(&mutex);
    contador++;
    printf("Thread %d, contador %d\n", pthread_self(), contador);
    pthread_mutex_unlock(&mutex);
  }
  return NULL;
}

int main() {
  pthread_t threads[NTHREADS];
  pthread_mutex_init(&mutex, NULL);
  for (int i = 0; i < NTHREADS; i++)
    pthread_create(&threads[i], NULL, incrementa, NULL);
  for (int i = 0; i < NTHREADS; i++)
    pthread_join(threads[i], NULL);
  pthread_mutex_destroy(&mutex);
  printf("Valor final do contador: %ld\n", contador);
  return 0;
}
#+end_src

* Variáveis de Condição

- Mecanismo de sincronização entre threads
- Permite esperar até que uma condição lógica seja verdadeira
- Sempre usado em conjunto com mutex
- Evita a espera ativa totalmente

#+latex: \vfill\pause

** Funcionamento

1. Thread adquire o mutex
2. Verifica uma condição do programa
3. Se falsa → espera na variável de condição
   - =pthread_cond_wait= destrava atomicamente o mutex que está segurando
4. Outra thread sinaliza a condição
5. Thread acorda e reavalia a condição

#+latex: \vfill\pause

** Observações

- Variáveis de condição não têm memória
  - Se um sinal é enviado sem thread esperando, o sinal é perdido
  - Programadores devem ser cuidadosos para não perder sinais 
- No passo 3, usar while e não if \to Evita problemas de sincronização

* Variáveis de Condição em Pthreads

Pthreads oferecem variáveis de condição além dos mutexes

#+latex: \vfill

| Chamada de thread        | Descrição                                             |
|--------------------------+-------------------------------------------------------|
| =pthread_cond_init=        | Cria uma variável de condição                         |
| =pthread_cond_destroy=     | Destrói uma variável de condição                      |
| =pthread_cond_wait=        | É bloqueado esperando por um sinal                    |
| =pthread_cond_signal=      | Sinaliza para outra thread e a desperta               |
| =pthread_cond_broadcast=   | Sinaliza para múltiplas threads e desperta todas      |

#+latex: \vfill

- =pthread_cond_wait=: bloqueia atomicamente liberando o mutex
- =pthread_cond_broadcast=: acorda todas as múltiplas threads bloqueadas
  - Por isso devemos usar o =while=, a condição pode ter mudado

* Exemplo de Produtor-Consumidor com Mutex e Condição

#+begin_src C :tangle aula09-pc-mutex-cond.c
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>
#define MAX 100
pthread_mutex_t the_mutex;
pthread_cond_t condc, condp;
int buffer = 0;

void *producer(void *ptr) {
  int i;
  for (i = 1; i <= MAX; i++) {
    pthread_mutex_lock(&the_mutex);
    while (buffer != 0) pthread_cond_wait(&condp, &the_mutex);
    buffer = i;
    pthread_cond_signal(&condc);
    pthread_mutex_unlock(&the_mutex);
  }
  pthread_exit(NULL);
}
void *consumer(void *ptr) {
  int i;
  for (i = 1; i <= MAX; i++) {
    pthread_mutex_lock(&the_mutex);
    while (buffer == 0) pthread_cond_wait(&condc, &the_mutex);
    buffer = 0;
    pthread_cond_signal(&condp);
    pthread_mutex_unlock(&the_mutex);
  }
  pthread_exit(NULL);
}

int main() {
    pthread_t prod, cons;

    pthread_mutex_init(&the_mutex, NULL);
    pthread_cond_init(&condc, NULL);
    pthread_cond_init(&condp, NULL);

    pthread_create(&prod, NULL, producer, NULL);
    pthread_create(&cons, NULL, consumer, NULL);

    pthread_join(prod, NULL);
    pthread_join(cons, NULL);

    pthread_mutex_destroy(&the_mutex);
    pthread_cond_destroy(&condc);
    pthread_cond_destroy(&condp);

    return 0;
}
#+end_src

* Semáforos

Um semáforo S é
- é essencialmente um contador inteiro protegido (p/ controle de acesso)
- acessado por duas operações atômicas (incremento, decremento)
- tem estado próprio (o contador)

#+latex: \vfill

#+begin_src C
semaphore_t S = 0;

wait(S) {   // ou down, ou P (proberen)
  while (S <= 0); /* espera ativa */
  S--;
}
signal(S) { // ou up, ou V (verhogen)
  S++;
}
#+end_src

#+latex: \vfill

- =wait()= era chamada P (/proberen/, ``testar'' em holandês)
- =signal()= era chamada V (/verhogen/, ``incrementar'' em holandês)
- Tanenbaum usa os nomes =down= e =up= (generalizações de =sleep= e =wakeup=)
- Todas as modificações em =wait()= e =signal()= devem ser indivisíveis
- Um semáforo com valor 0 indica que nenhum sinal de despertar foi salvo

* Uso de Semáforos

Dois tipos: semáforos de contagem e semáforos binários

#+latex: \vfill

Semáforo de contagem:

- Valor pode variar num domínio irrestrito
- Controla acesso a recurso com número finito de instâncias
- Inicializado com o número de recursos disponíveis

#+latex: \vfill

Semáforo binário:

- Valor varia somente entre 0 e 1
- Comporta-se de modo semelhante ao lock mutex

#+latex: \vfill

Exemplo: usar semáforos para impor uma sincronização
- garantir que S2 execute somente após S1 (semáforo =synch= init. 0)

#+begin_src C
// P1:  S1; signal(synch);
// P2:  wait(synch); S2;
#+end_src

* Implementação de Semáforo 1/2

Para eliminar a espera ativa, o processo bloqueia a si próprio

- Op. =block()= insere o processo em fila de espera associada ao semáforo
- Estado do processo é comutado para espera; controle vai ao escalonador
- Op. =wakeup(P)= reinicia P: do estado de espera para o estado de pronto

#+latex: \vfill

#+begin_src C
typedef struct {
  int value;
  struct process *list;
} semaphore;

wait(semaphore *S) {
  S->value--;
  if (S->value < 0) {
    adiciona esse processo a S->list;
    block();
  }
}
signal(semaphore *S) {
  S->value++;
  if (S->value <= 0) {
    remove um processo P de S->list;
    wakeup(P);
  }
}
#+end_src

* Implementação de Semáforo 2/2

=wait()= e =signal()= devem ser executadas atomicamente

#+latex: \vfill

Em ambiente com um único processador:

- Inibir interrupções durante a execução de =wait()= e =signal()=
- Sem interrupções, instruções de processos não são intercaladas

#+latex: \vfill

Em ambiente multiprocessador:

- Interrupções devem ser desabilitadas em cada processador
  - Difícil e prejudicial ao desempenho
- Sistemas SMP usam técnicas alt.: =compare_and_swap()= ou spinlocks

#+latex: \vfill\pause

- A espera ativa foi apenas movida para as seções de =wait()= e =signal()=
  - Essas seções são curtas
- Muito diferente de seções críticas longas (minutos ou horas)

* Resolvendo o Problema do Dormir e Acordar

Semáforos solucionam o problema do sinal de despertar perdido

#+latex: \vfill

#+begin_src C
#define N 100
typedef int semaphore;
semaphore mutex = 1; /* controla acesso a regiao critica */
semaphore empty = N; /* conta lugares vazios no buffer */
semaphore full  = 0; /* conta lugares preenchidos */

void producer(void) {
  int item;
  while (TRUE) {
    item = produce_item();
    wait(&empty);       /* decresce contador empty */
    wait(&mutex);       /* entra na regiao critica */
    insert_item(item);
    signal(&mutex);     /* sai da regiao critica */
    signal(&full);      /* incrementa lugares preenchidos */
  }
}
void consumer(void) {
  int item;
  while (TRUE) {
    wait(&full);        /* decresce contador full */
    wait(&mutex);       /* entra na regiao critica */
    item = remove_item();
    signal(&mutex);     /* sai da regiao critica */
    signal(&empty);     /* incrementa lugares vazios */
    consume_item(item);
  }
}
#+end_src

* Análise da solução em Pthreads

Semáforos são usados de duas maneiras distintas no produtor-consumidor:

- Exclusão mútua: =mutex= garante que apenas um processo acessa o buffer
- Sincronização: =full= e =empty= garantem sequências de execução corretas

* Produtor-Consumidor com POSIX Semáforos 1/2
** Funções básicas
#+begin_src C :tangle aula09-pc-sem.c :main no
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>
#include <unistd.h>
#define N 100
int buffer[N];
int in = 0;
int out = 0;

int produce_item() {
  int item = rand() % 1000;
  usleep(1000000); //custo de produção: 1s
  printf("%lu Produzido: %d\n", pthread_self(), item);
  return item;
}
void insert_item(int item) {
  buffer[in] = item;
  in = (in + 1) % N;
}
int remove_item() {
  int item = buffer[out];
  out = (out + 1) % N;
  return item;
}
void consume_item(int item) {
  usleep(10000);  //custo de consumo: 0.1s
  printf("%lu Consumido: %d\n", pthread_self(), item);
}
#+end_src

* Problemas com Semáforos

A ordem das operações com semáforos é crítica

#+latex: \vfill

Exemplo: se no produtor =wait(&mutex)= vier antes de =wait(&empty)=

- Se o buffer estiver cheio, produtor bloqueia com =mutex = 0=
- Consumidor tenta =wait(&mutex)= e também bloqueia
- Ambos ficam bloqueados para sempre: impasse (deadlock)

#+begin_src C
void producer(void) {
  int item;
  while (TRUE) {
    item = produce_item();
    wait(&empty);       /* decresce contador empty */
    wait(&mutex);       /* entra na regiao critica */
    insert_item(item);
    signal(&mutex);     /* sai da regiao critica */
    signal(&full);      /* incrementa lugares preenchidos */
  }
}
#+end_src

#+latex: \vfill

- Erros com semáforos \to programação em ling. de montagem, mas piores
- Condições de corrida, Impasses, Comportamentos irreprodutíveis

#+latex: \vfill\pause

** Proposta de Brinch Hansen e Hoare: monitores

- Coleção de rotinas, variáveis e estruturas de dados em módulo especial
- Apenas um processo pode estar ativo num monitor em qualquer instante
- O compilador implementa a exclusão mútua: menos propenso a erros

* Deadlocks e Inanição                                             :noexport:

Implementação de semáforo com fila de espera pode resultar em deadlock

#+latex: \vfill

Exemplo com dois processos (=S= e =Q= inicializados com 1):

#+begin_src C
// P0:  wait(S); wait(Q); ... signal(S); signal(Q);
// P1:  wait(Q); wait(S); ... signal(Q); signal(S);
#+end_src

- P0 executa =wait(S)=, P1 executa =wait(Q)=
- P0 espera por =signal(Q)= de P1; P1 espera por =signal(S)= de P0
- Nenhum sinal pode ser executado: deadlock

#+latex: \vfill

Um conjunto de processos está em deadlock quando cada um espera por evento

- que só pode ser causado por outro processo do mesmo conjunto

#+latex: \vfill

Inanição (/starvation/): processos esperam indefinidamente dentro do semáforo

- Pode ocorrer se a remoção da fila de espera seguir ordem LIFO

* Inversão de Prioridades                                          :noexport:

Desafio de escalonamento quando processo de alta prioridade precisa de lock

#+latex: \vfill

** Exemplo com três processos (prioridade L < M < H):

- H precisa do recurso R, que está sendo usado por L
- M torna-se executável e causa preempção de L
- H fica aguardando, indiretamente bloqueado por M (prioridade menor)

#+latex: \vfill\pause

** Solução: protocolo de herança de prioridades

- L herda temporariamente a prioridade de H enquanto usa R
- M não consegue causar preempção de L (mesma prioridade de H)
- Ao liberar R, L retorna à sua prioridade original
- H executa imediatamente após a liberação

#+latex: \vfill\pause

** Caso real: Mars Pathfinder (1997)

- Tarefa H =bc_dist= bloqueada por =ASI/MET= (tarefa L)
- =ASI/MET= sofria preempção por tarefas de prioridade média
- Solução: habilitar herança de prioridades nos semáforos do VxWorks

* O Problema do Jantar dos Filósofos 1/3

Cinco filósofos sentados em torno de uma mesa circular com espaguete

- Cada filósofo precisa de dois garfos para comer
- Entre cada par de pratos há um garfo (cinco garfos no total)
- A vida de um filósofo: alternar períodos de alimentação e pensamento

#+latex: \vfill

Solução ingênua \to contém deadlock:

#+begin_src C
#define N 5
void philosopher(int i) {
  while (TRUE) {
    think();
    take_fork(i);           /* pega o garfo esquerdo */
    take_fork((i+1) % N);  /* pega o garfo direito */
    eat();
    put_fork(i);
    put_fork((i+1) % N);
  }
}
#+end_src

- Se todos pegam o garfo esquerdo simultaneamente, ninguém pega o direito

* O Problema do Jantar dos Filósofos 2/3

Tentativa de correção: devolver o garfo esquerdo se o direito não estiver livre

#+latex: \vfill

- Ainda pode falhar: todos pegam, devolvem e repetem indefinidamente
- Situação em que todos executam mas nenhum progride: inanição (/starvation/)

#+latex: \vfill

** Tentativa com semáforo binário global:

- Antes de pegar garfos, faz =wait(&mutex)=; ao largar, =signal(&mutex)=
- Solução correta do ponto de vista teórico
- Problema de desempenho: apenas um filósofo pode comer por vez
- Com cinco garfos, dois filósofos podem comer simultaneamente

* O Problema do Jantar dos Filósofos 3/3
#+begin_src C :tangle aula09-filosofos.c
#define N 5
#define LEFT  (i + N - 1) % N
#define RIGHT (i + 1) % N
typedef enum { Thinking, Hungry, Eating } Estado;
typedef int semaphore;
Estado state[N] = {0};
semaphore mutex = 1;
semaphore s[N]; /* um semaforo por filosofo */
void take_forks(int i) {
  wait(&mutex);      /* trava da região crítica: entra */
  state[i] = Hungry;
  test(i);           /* tenta pegar dois garfos */
  signal(&mutex);    /* trava da região crítica: sai */
  wait(&s[i]);       /* bloqueia se garfos nao disponiveis */
}
void put_forks(int i) {
  wait(&mutex);      /* trava da região crítica: entra */
  state[i] = Thinking;
  test(LEFT);        /* ve se vizinho esquerdo pode comer */
  test(RIGHT);       /* ve se vizinho direito pode comer */
  signal(&mutex);    /* trava da região crítica: sai */
}
void test(int i) {
  /* se i está faminto e os vizinhos não estão comendo */
  if (state[i] == Hungry && state[LEFT] != Eating && state[RIGHT] != Eating) {
    state[i] = Eating;
    signal(&s[i]);
  }
}
#+end_src

* O Problema dos Leitores e Escritores                             :noexport:

Modela acesso a banco de dados com múltiplos leitores e escritores

- Múltiplos processos podem ler simultaneamente
- Escritor precisa de acesso exclusivo (nem leitores, nem outros escritores)

#+latex: \vfill

#+begin_src C
typedef int semaphore;
semaphore mutex = 1; /* controla acesso a 'rc' */
semaphore db    = 1; /* controla acesso a base de dados */
int rc = 0;          /* numero de leitores ativos */

void reader(void) {
  while (TRUE) {
    down(&mutex); rc++; if (rc == 1) down(&db); up(&mutex);
    read_data_base();
    down(&mutex); rc--; if (rc == 0) up(&db);   up(&mutex);
    use_data_read();
  }
}
void writer(void) {
  while (TRUE) {
    think_up_data();
    down(&db);
    write_data_base();
    up(&db);
  }
}
#+end_src

- O primeiro leitor bloqueia escritores; o último libera
- Problema: leitores contínuos podem causar inanição do escritor
