* Introdução a processos

*Processo* é a instância de um programa em execução

- Inclui: contador de programa, registradores, variáveis
  - _Programa_, _Entrada_, _Saída_, _Estado_ 
- Cada processo tem sua própria CPU virtual
- A CPU real alterna entre processos rapidamente
- Esse mecanismo é chamado *multiprogramação*

#+latex: \vfill\pause

*Programa × Processo*

- Programa: entidade *passiva* (arquivo em disco)
- Processo: entidade *ativa* (programa carregado na memória)
- O mesmo programa pode originar múltiplos processos

* Criação de processos

# - Fonte: Tanenbaum & Bos — Cap. 2, Sec. 2.1.2

Quatro eventos que causam a criação de processos

- Inicialização do sistema (processo =init=)
- Chamada de sistema por um processo em execução
- Solicitação de um usuário para criar um novo processo
  - Cliques em ícones, execução via terminal
- Início de uma tarefa em lote (=sbatch= no PCAD)

#+latex: \vfill

*Daemons*: processos em segundo plano que gerenciam serviços do sistema

- Exemplos: e-mail, web, impressão
- No UNIX, o comando =ps= lista os processos em execução

#+latex: \vfill

- Um processo pode criar filhos para auxiliar no trabalho
  - Podemos dividir uma carga de trabalho entre os processos filhos

* Criação de processos na prática

# - Fonte: Tanenbaum & Bos — Cap. 2, Sec. 2.1.2; Silberschatz — Cap. 3, Sec. 3.3

*UNIX*: chamada de sistema =fork=

- =fork=: cria um clone exato do processo pai
- O filho chama =execve= (ou similar) para carregar novo programa
# - Dois passos permitem redirecionar stdin/stdout/stderr antes do exec

#+latex: \vfill

# IExemplo em C (Silberschatz — Figura 3.9):

#+BEGIN_SRC C
pid_t pid = fork();
if (pid == 0) {       /* filho */
    execlp("/bin/ls","ls",NULL);
} else {              /* pai */
    wait(NULL);
}
#+END_SRC

#+latex: \vfill\pause

*Windows*: chamada =CreateProcess=

- Cria o processo e carrega o programa em uma única chamada
- Mais de dez parâmetros de configuração

** Programa para criar processos com fork/execve                  :noexport:

#+begin_src C :tangle fork.c
#include <sys/types.h>
#include <sys/wait.h>
#include <stdio.h>
#include <unistd.h>
int main(int argc, char **argv)
{
  pid_t pid = 0;
  /* cria um processo-filho */
  pid = fork();
  if (pid < 0) { /* um erro ocorreu */
    fprintf(stderr, "Fork Failed.\n");
    return 1;
  }else if (pid == 0) { /* processo-filho */
    execve("/bin/ls", argv, NULL);
  }else { /* processo-pai */
    /* o pai esperará que o filho seja concluído */
    printf("Parent is waiting.\n");
    wait(NULL);
    printf("Child Complete.\n");
  }
  return 0;
}
#+end_src

* O que acontece depois do =fork()=

# - Fonte: Tanenbaum & Bos — Cap. 2, Sec. 2.1.2; Silberschatz — Cap. 3, Sec. 3.3

Após =fork=, pai e filho possuem *espaços de endereços (virtual) distintos*
- Nenhuma memória gravável é compartilhada entre eles
- Alterações de variáveis (estáticas, globais) em um não afetam o outro

#+latex: \vfill

Recursos que *podem* ser compartilhados
- Arquivos abertos herdados do pai
- Descritores de arquivo (antes do =exec=)

#+latex: \vfill

Duas abordagens possíveis após a criação
- *Pai continua* executando concorrentemente ao filho
- *Pai espera* (=wait=) até que o filho termine

* Término de processos

# - Fonte: Tanenbaum & Bos — Cap. 2, Sec. 2.1.3; Silberschatz — Cap. 3, Sec. 3.3

Quatro condições de término
1. Saída normal: voluntária (=exit=)
2. Saída por erro: voluntária (erro detectado pelo processo)
3. Erro fatal: involuntário (instrução ilegal, divisão por zero)
4. Morto por outro processo: involuntário (=kill= / =TerminateProcess=)

#+latex: \vfill\pause
   
*Processo zumbi*
- Processo encerrado cujo pai ainda não chamou =wait()=
- Entrada na tabela de processos permanece até o pai chamar =wait()=
- Contém o status de saída do processo

#+latex: \vfill\pause

*Encerramento em cascata*
- Alguns sistemas não permitem filho sem pai
- Se o pai termina, todos os filhos são encerrados
  - Muitas vezes isso é algo opcional (=gerenciador de login=)
- Isso nos remete a hierarquia de processos =Unix=
    
* Hierarquia de processos

# - Fonte: Tanenbaum & Bos — Cap. 2, Sec. 2.1.2

*UNIX*: hierarquia de processos (=pstree -p=)
- Um processo filho pode criar mais processos
- Processo + todos os descendentes = *grupo de processos*
- Todos os processos pertencem a uma única árvore
  - Raiz da árvore: processo =init=
- Processos UNIX não podem repassar (deserdar) seus filhos

#+latex: \vfill\pause

*Windows*: sem hierarquia formal
- Pai recebe um *handle* (identificador) ao criar um filho
  - O handle pode ser passado para outro processo
  - Isso invalida qualquer noção de hierarquia
- Não há equivalente ao =init=

*Escalonador* (que implementa a _multiprogramação_)
- Tratamento de interrupções, maximizando o uso da CPU
- Sempre ter algum processo em execução
  - Alternar a CPU entre processos com frequência
  - Usuários podem interagir com cada programa

* O tratamento de interrupção

# - Fonte: Tanenbaum & Bos — Cap. 2, Sec. 2.1.3; Silberschatz — Cap. 3, Sec. 3.2

A cada interrupção de HW (preempção ou E/S) ou de SW (programa)
1. O HW empilha o PC (/Program Counter/) e outras informações
2. O HW carrega o PC a partir do _vetor de interrupções_
3. O _vetor de interrupções_ em /assembly/ salva os registradores
4. O procedimento em /assembly/ configura uma nova pilha
5. O serviço de interrupção em C executa
6. O *escalonador* decide qual processo é o próximo a executar
7. O procedimento em C retorna para o código em /assembly/
8. O procedimento em /assembly/ inicia o novo processo atual

#+latex: \pause\vfill

Precisamos manter informações (PC, etc) sobre cada processo
- Cada processo possui um PCB (/Process Control Block/) -- é uma =struct=
  - Existe uma ``tabela'' de todos os PCBs (normalmente uma lista)

* Bloco de Controle de Processo (PCB)

# - Fonte: Tanenbaum & Bos — Cap. 2, Sec. 2.1.4; Silberschatz — Cap. 3, Sec. 3.1.3

Campos do PCB
- Estado do processo (novo, pronto, executando…)
- Contador do programa (próxima instrução)
- Registradores da CPU (acumuladores, pilha, índice)
- Informações de scheduling (prioridade, ponteiros de filas)
- Informações de gerenciamento de memória

#+latex: \vfill\pause
  
Salvo/restaurado a cada troca de estado
- Quando o processo sai da execução
- Para estados pronto ou bloqueado
- _Permite retomar como se nunca tivesse parado_

#+latex: \vfill\pause

*Mudança de contexto* (context switch)
- Kernel salva PCB do processo corrente
- Kernel carrega PCB do próximo processo
- Tempo gasto é de sobrecarga do SO (sem trabalho útil)
  - Velocidade típica: alguns milissegundos

* Modelando a multiprogramação

- Obj: Estimar a utilização da CPU em função do grau de multiprogramação
- Seja $p$ a fração do tempo que um processo fica em E/S
  - Com $n$ processos, a probabilidade de *todos* esperando por E/S é $p^n$
- Assim, a métrica ``Utilização da CPU'' \to \text{Utilização} = 1 - p^n

#+latex: \pause\vfill

#+begin_src R :results file output graphics :file img/modelando-multiprogramacao.pdf :exports results :width 6 :height 2 :session *R*
options(crayon.enabled=FALSE)
library(tidyverse)
crossing(p = c(0.2, 0.5, 0.8), n = 0:20) |>
  mutate(U = 1 - p^n) |>
  mutate(p = as.factor(p)) |>
  ggplot(aes(x = n, y = U, color = p, group = p)) +
  geom_line(alpha=.8) +
  geom_point() +
  theme_bw(base_size=12) +
  scale_color_brewer(palette = "Set1", name = "Group") +
  theme(
    plot.margin = unit(c(0, 0, 0, 0), "cm"),
    legend.spacing = unit(1, "mm"),
    panel.grid = element_blank(),
    legend.position = "top",
    legend.justification = "left",
    legend.box.spacing = unit(0, "pt"),
    legend.box.margin = margin(0, 0, 0, 0)) +
  ylab("Utilização de CPU [%]") +
  xlab("Quantidade de processos na memória")
#+end_src
